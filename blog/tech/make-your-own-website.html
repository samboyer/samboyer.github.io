How to make your own website for free with GitHub Pages and your own HTML!

Centralised Internet Platforms Hate Him!

February 2024

Sam B

{intro}

Warning: while this is a beginner-friendly tutorial, the intent is to teach you a bit about how the Web works. If you don’t care about technology and just want to write stuff (which is fine!!), you might be better off with a platform like Wordpress.com or Blogger. (Both have export features, so you should be able to move your stuff to a new site if necessary.)

What I’ll be covering in this post:

  • Creating a GitHub Pages repository
  • (optionally) Registering a custom domain and linking it to your Pages site
  • Adding content to your site via git

What I won’t be covering: - Absolutely everything there is to know about HTML/CSS - Generating HTML from text/Markdown (static site generators)

Step 1: Make a GitHub Pages repository and put something in it.

  1. If you don’t have one already, create a GitHub account here.

  2. Once you’re logged in, create a new repository here, giving it the name 'YOUR_GITHUB_USERNAME.github.io'. All the other settings can be left at their defaults.

  3. Now we’ll clone the repository to your computer, so you can add files to it.

    1. If you’ve never use git before, I’d recommend GitHub Desktop or SourceTree, both available on Windows and OSX.

    2. From your repository page on GitHub, click the green ‘<> Code’ button. If you’re using GitHub Desktop, the ‘Open in GitHub Desktop’ link should work; otherwise you’ll need to copy the 'git@github.com/....git' URL into your client. CLI users can run git clone git@github.com/...git.

  4. Create a file in your ‘checked out’ repository called ‘index.html’. Add some text like ‘Hello world!’.

  5. Use your Git client to Commit and Push the file (sometimes called Sync). When you reload the GitHub repository page, your new file should appear there - and even better, navigating to YOUR_GITHUB_USERNAME.github.io in the web browser should show that message you just wrote!

(Optional) Get yourself a domain name

By default, GitHub Pages are accessible on the web via 'yourname.github.io'. While this is a fine option for code projects and the like, it has the following disadvantages:

  • You won’t be able to move the website off GitHub in the future
  • You won’t be able to register any subdomains (e.g. ), or use any other DNS features (like MX records for mail servers).
  • You won’t be impressing any hipster domain connoisseurs with ‘.github.io’ in your domain name. (whether you care about that is up to you.)

When choosing a name:

  • Make sure it’s one you’ll be happy with in few years time. Changing a domain after the fact is a bit of a pain, and you’ll likely forget to update the URL in a few places.
  • Be careful with unconventional TLDs, especially when they’re owned by less-savoury countries/organisations. It’s fun to combine domain names with extensions to make a word (e.g. youtu.be), or jump on the whichever TLD is trendy this year, but this can and does result in unexpected domain suspensions, or at the very least getting blocked by lazy domain-level firewalls.

(Optional) Use a static site generator.

Static site generators (Jekyll, Hugo etc) take in ‘preformatted text’ (think Markdown or .txt files) and output HTML. While this can be great for speeding up the process of creating webpages (this blog uses an SSG), they’re not beginner friendly, and don’t make sense for a website with few pages. Florens Verschelde summarizes the topic well here.

If you’re comfortable with writing Markdown, and running commands from a terminal, and plan to add lots of text to your website, then a SSG might be worthwhile. In that case, Jekyll might be your best bet - GitHub Pages can build your site from source each time you push to the main branch. Otherwise, I’d recommend keeping a seperate branch/repository containing your preformatted content, and writing some precommit hook to commit the generated pages to the ‘production’ branch (usually ‘main’, but this can be customised in the GitHub Pages settings).

Conclusion

By now you should have a working website, either living under github.io or your own snazzy domain name. Whatever you add now is up to you!
